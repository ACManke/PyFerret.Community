On Wed, Dec 16, 2015 at 2:30 AM, Ansley C. Manke
<ansley.b.manke@xxxxxxxx> wrote:
> Hi,
> Yes, we haven't done much development with on binary files, as the demand
> for them seems to be pretty low ever since NetCDF took over as the dominant
> file type for most users.  The precision written just goes along with the
> internal stored data type which has been double precision since Ferret v6.8.

But, THAT is an important piece of information.  I'm not demanding
immediate fixes.  I just need to know what the present status is so
that I can come up with a RELIABLE workaround.  So, the information
that the "precision written just goes along with the
internal stored data type which has been double precision since Ferret
v6.8" is important, because I was wondering if that was unique to, or
even a bug of,  the Mac version of Ferret.

Also, maybe can we expect that OUTTYPE will be extended to stream and
unformatted binaries in the future?

As an aside, one big downside of netCDF is that its Fortran-90 API is
very user-UNfriendly.  (Its C and Fortran-77 API is even worse.)  I've
written a lot of Fortran code to read and write netCDF files, but each
time I find it painful.  I think that's one of the reasons why people
generally stick to stream binary in the GrADS community.  (Ferret can
do a lot more data analysis than GrADS and so fewer people need
Fortran in the Ferret community.)

Right now, I have to use stream binary on Ferret because I'm
collaborating with a colleague who uses GrADS and Fortran.
